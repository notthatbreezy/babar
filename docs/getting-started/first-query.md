# Your first query

In this chapter we'll connect to Postgres, define a tiny authored schema,
insert one row, and read it back through babar's default typed SQL surface:
schema-aware `query!` / `command!`.

## Setup

Add `babar` and a Tokio runtime to your `Cargo.toml`, then drop the
following into `src/main.rs`.

```rust
use babar::query::{Command, Query};
use babar::{Config, Session};

babar::schema! {
    mod app_schema {
        table demo_users {
            id: primary_key(int4),
            name: text,
        }
    }
}

#[tokio::main(flavor = "current_thread")]
async fn main() -> babar::Result<()> {
    // 1. Describe the connection.
    let cfg = Config::new("localhost", 5432, "postgres", "postgres")
        .password("postgres")
        .application_name("first-query");

    // 2. Open a Session. The Session owns one Postgres connection.
    let session: Session = Session::connect(cfg).await?; // type: Session

    // 3. DDL is outside the typed SQL subset, so use an explicit raw fallback.
    let create: Command<()> = Command::raw(
        "CREATE TEMP TABLE demo_users (id int4 PRIMARY KEY, name text NOT NULL)",
        (),
    );
    session.execute(&create, ()).await?;

    // 4. Application SQL uses the schema-aware wrappers generated by schema!.
    let insert: Command<(i32, String)> =
        app_schema::command!(INSERT INTO demo_users (id, name) VALUES ($id, $name));
    session.execute(&insert, (1, "Ada".to_string())).await?;

    let q: Query<(i32,), (i32, String)> = app_schema::query!( // type: Query<(i32,), (i32, String)>
        SELECT demo_users.id, demo_users.name
        FROM demo_users
        WHERE demo_users.id = $id
    );

    // 5. Run it. `query` returns Vec<B> — one decoded tuple per row.
    let rows: Vec<(i32, String)> = session.query(&q, (1,)).await?; // type: Vec<(i32, String)>

    for (id, name) in &rows {
        println!("id={id} name={name}");
    }

    session.close().await?;
    Ok(())
}
```

Run it with a Postgres reachable on `localhost:5432`:

```text
cargo run
# id=1 name=Ada
```

## Breaking this down

**`Config::new(host, port, user, database)`** takes the four required fields by
position. Optional fields are chained on after:
`.password(...)`, `.application_name(...)`, `.connect_timeout(...)`, and so on.
There is no `Config::from_env()` and no DSN parser — `Config` is a plain struct.

**`Session::connect(cfg)`** returns a `Session`. A `Session` owns one Postgres
connection plus a background task that owns the socket. Every method you call on
`Session` is cancellation-safe: dropping the future won't leave the connection
half-spoken-to.

**`babar::schema! { ... }`** defines authored schema facts in Rust and generates
schema-scoped wrappers like `app_schema::query!(...)` and
`app_schema::command!(...)`. That is the recommended reusable pattern in v0.1.
For one-off examples or tests you can also keep schema facts inline with
`babar::query!(schema = { ... }, SELECT ...)`.

**`Query<(i32,), (i32, String)>`** is still the heart of the typed surface. The
first type parameter is the bound parameter tuple, the second is the decoded row
shape. `query!` infers both from the authored schema and the SQL you wrote.

**`query!` / `command!`** are the default typed SQL path. They share one
schema-aware compiler, use named placeholders like `$id`, and can optionally
live-verify schema-aware `SELECT` statements against a database at macro
expansion time when `BABAR_DATABASE_URL` or `DATABASE_URL` is set. Today that
probe-verification path checks authored schema facts, parameters, and returned
columns for `query!` / `typed_query!` reads; `command!` shares the compiler but
is not yet probe-verified through that hook.

That surface is intentionally narrow:

- one statement at a time
- authored Rust schema only — no file-based schema input, codegen, or offline cache
- optional forms stay explicit: `$value?` and `(...)?` only in supported spots
- unsupported constructs fall back to raw SQL instead of trying to rewrite everything

**`typed_query!`** remains available as a compatibility alias to the same
compiler during the migration from older docs and examples, but new code should
prefer `query!` / `command!`.

**`Query::raw` / `Command::raw`** are the typed fallback when a statement is
outside the current typed SQL subset but you still want the extended protocol,
typed params, typed rows, prepare support, or streaming. Use
**`simple_query_raw`** only for simple-protocol raw SQL strings such as
multi-statement bootstrap or migration work where typed params/results are not
the goal.

**`session.query(&q, args)`** is the run step. It returns `Vec<B>` — fully
decoded rows, where each `B` is whatever your query type says it is. babar does
not expose an intermediate `Row` type and there is no `.get::<T, _>()`
accessor.

## What happened

You connected to PostgreSQL, used a raw fallback for setup DDL, inserted one row
through schema-aware typed SQL, fetched it back as `(i32, String)`, and closed
the session.
