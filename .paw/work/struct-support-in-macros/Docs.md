# Struct Support in Macros

## Overview

This work makes schema-aware typed SQL macros behave like the rest of `babar`'s
typed boundary: command inputs, query inputs, and query rows can now use named
Rust structs as the primary contract when the SQL shape is a named field set.
The macro path now documents and enforces the same struct-centric workflow that
raw builders and derived codecs already supported.

## Architecture and Design

### High-Level Architecture

The implementation extends the existing schema-aware typed SQL pipeline instead
of creating a separate runtime path. Public macro parsing now carries
struct-shape intent, validation checks the selected Rust type against resolved
SQL metadata, and lowering reuses the existing codec-based statement machinery.

### Design Decisions

- Schema-aware macros support both explicit selection (`params = Type`,
  `row = Type`) and surrounding-type inference (`params = _`, `row = _`).
- Explicit selections take precedence over surrounding `Query<A, B>` or
  `Command<A>` types so the macro site can pin the contract.
- Strict struct matching is intentional: missing fields, extra fields, and
  type/nullability mismatches are rejected instead of being tolerated.
- Query row structs match by final output names, including aliases, rather than
  projection order alone.
- Tuple-style behavior remains available when the SQL contract is not a strict
  named-struct contract.

### Integration Points

- `schema!` modules and inline `schema = { ... }` blocks share the same
  selection and validation rules.
- Row/parameter validation builds on the existing derived codec model instead of
  inventing a new runtime representation.
- Public docs now teach named structs as the normal schema-aware macro path and
  use one shared struct whenever input and output field sets are identical.

## User Guide

### Prerequisites

- Use schema-aware `query!` / `command!` calls, either through a `schema!` module
  or an inline `schema = { ... }` block.
- Derive `babar::Codec` on any struct used as a parameter or row shape.

### Basic Usage

- Let the surrounding statement type infer struct shapes when that keeps the code
  clear.
- Reuse one struct for both command input and query rows when the SQL field sets
  are identical.
- Split parameter and row structs only when the input placeholders and projected
  output names differ.

### Advanced Usage

- Add `params = Type` and `row = Type` to pin the selected structs directly at
  the macro call site.
- Add `params = _` and `row = _` when you want inference to remain explicit in
  the macro invocation.
- Use SQL aliases to match row fields by their final output names.

## API Reference

### Key Components

- Schema-scoped `query!` / `command!` wrappers generated by `schema!`
- Top-level `babar::query!` / `babar::command!` with inline schema
- Struct selection clauses: `params = Type`, `row = Type`, `params = _`,
  `row = _`

### Configuration Options

- Live query verification still depends on `BABAR_DATABASE_URL` or
  `DATABASE_URL` at macro expansion time.
- Feature-gated Postgres type families still require the corresponding Cargo
  feature when used in selected structs.

## Testing

### How to Test

- Build the docs book with `mdbook build`.
- Check workspace rustdoc with `RUSTDOCFLAGS="-D warnings" cargo doc --workspace --no-deps`.
- Run `cargo test --all-features`.
- For manual review, verify that schema-aware examples show shared structs only
  when input and output field sets match, and that explicit `params = Type` /
  `row = Type` examples describe alias-aware row matching.

### Edge Cases

- Duplicate or ambiguous final output names reject strict row-struct selection.
- Output aliases control row-field matching names.
- Nullable SQL fields still require matching `Option<T>`-style Rust fields.
- Extra fields on either input or row structs are rejected.

## Limitations and Future Work

- Dynamic typed SQL statements still reject `params = ...` selection.
- Tuple fallback remains necessary for SQL contracts that are not naturally a
  strict named struct.
- The feature does not broaden the supported typed-SQL subset beyond the
  struct-selection and validation work shipped here.
